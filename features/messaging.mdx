## Overview

Messaging is the core interaction loop. Since V16.8, the pipeline uses **conversation orchestration** — messages are classified, collaboration plans are built, runs are queued with dependencies, and the client processes them asynchronously.

## Client component

`RoomChat` (`src/components/RoomChat.tsx`) handles:

- Message composition via `ChatComposer`
- @mention autocomplete (ordered mentions for collaboration)
- Collaboration plan banner (`CollaborationPlanBanner`)
- Async run processing with polling recovery
- Optimistic UI updates

## Send message flow

### 1. Client sends message

```typescript
POST /api/rooms/{roomId}/messages
Authorization: Bearer {access_token}

{
  "content": "@PM work with @Engineer on the launch plan",
  "topicId": "uuid",
  "mentionsJson": [
    { "type": "ai_employee", "id": "pm-1", "name": "PM" },
    { "type": "ai_employee", "id": "eng-1", "name": "Engineer" }
  ]
}
```

### 2. Server saves and plans (parallel)

The route handler:

1. Validates auth, room membership, topic access
2. **In parallel:** inserts human message + loads responder context (V16.2 optimization)
3. Calls `planConversation()` — classifies mode, picks responders
4. Queues runs via `queueAgentRuns()` (with `depends_on_run_id` for collaborators)
5. Returns immediately — **no inline AI replies**

```json
{
  "humanMessage": {},
  "queuedRuns": [],
  "blockedRuns": [],
  "collaborationPlan": {
    "mode": "lead_collaborator",
    "participants": []
  },
  "aiMessages": []
}
```

### 3. Client polls and processes

```typescript
// Poll for processable runs
GET /api/rooms/{roomId}/topics/{topicId}/agent-runs?status=queued,waiting,running

// Process each run
POST /api/agent-runs/{runId}/process
{ "workspaceId": "uuid", "mode": "live" }
```

Collaborator runs in `waiting` status become processable when their `depends_on_run_id` run completes.

### 4. Realtime sync

Other tabs receive AI messages via Supabase Realtime on the `messages` table.

## Message artifacts (V16.2)

AI messages can include inline work objects:

```typescript
type MessageArtifact = {
  type: "task" | "memory" | "approval" | "work_log" | "email_draft";
  id: string;
  label: string;
};
```

Rendered by `RoomMessageItem` as clickable cards below the message text.

## AI reply sanitization (V16.1)

Structured LLM output is normalized before display:

- `normalize-model-response.ts` — extract conversational text, strip JSON wrappers
- `sanitize-effects.ts` — clean side-effect payloads (V16.8)
- DMs receive human-like tone via updated prompts

Raw JSON no longer leaks into chat.

## Error handling

| Code | Meaning |
|------|---------|
| `message_required` | Empty content |
| `topic_archived` | Cannot post to archived topic |
| `ai_runtime_failed_but_message_saved` | Message saved, AI queue failed (HTTP 207) |

## Topic AI controls

If `aiStopped` is true on the topic, no runs are queued regardless of mentions. See [Channel orchestration](/features/channel-orchestration).

## Demo mode fallback

When `backend === "demo"`, messages stay in-memory and `employee-engine.ts` produces scripted replies.
